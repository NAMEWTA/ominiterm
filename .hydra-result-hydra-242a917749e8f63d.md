High: [`electron/process-detector.ts:95`](/Users/zzzz/termcanvas/electron/process-detector.ts#L95) does not actually preserve BFS or "shallowest-first" ordering, even though [`electron/process-detector.ts:49`](/Users/zzzz/termcanvas/electron/process-detector.ts#L49) and the new test description say it does. The code uses BFS only to populate `descendants`, then it emits matches by iterating the original `ps` rows at [`electron/process-detector.ts:97`](/Users/zzzz/termcanvas/electron/process-detector.ts#L97). On macOS, `man ps` documents the default sort as controlling terminal and then PID, not tree depth. Because [`electron/process-detector.ts:125`](/Users/zzzz/termcanvas/electron/process-detector.ts#L125) returns `results[0]`, a deeper descendant can still win over a shallower/direct child if the shallow match started later. A concrete case is `shell -> bash -> codex` first, then a direct `claude` child launched afterwards: `ps` order will still surface the older deep `codex` first. The fundamental descendant traversal fix is right, but result ordering needs to come from the BFS traversal itself or from an explicit depth sort. The current regression test at [`tests/process-detector.test.ts:93`](/Users/zzzz/termcanvas/tests/process-detector.test.ts#L93) does not catch this because it only asserts `cliType`, not the winning PID, and it does not include competing matches at different depths.

## Review notes
- Descendant traversal is the right fix. The direct-child-only approach could never handle `volta`, `mise`, `npx`, shell wrappers, or similar launch chains reliably.
- Performance is fine. Parsing once, building a parent map, and traversing one subtree is linear in the `ps` snapshot size and should be cheap for this use case.
- A fixed depth bound is probably the wrong tradeoff. Wrapper depth is environment-dependent, so a bound would reintroduce false negatives. If a guard is wanted, cap total visited nodes rather than depth.
- Cycle safety is effectively okay for malformed input because `descendants` deduplicates visited PIDs. Seeding the visited set with `shellPids` would make that invariant clearer, but I do not see a real infinite-loop risk in normal `ps` output.
- False-positive risk is still driven more by `matchCli()` heuristics than by BFS itself. Wrapper matching scans the full remaining argv, so processes like `node tools/codex-migration.js` can still look like a Codex CLI. Descendant traversal widens the reach of that pre-existing heuristic, so it is worth monitoring, but I would treat it as a separate precision problem rather than a reason to reject subtree traversal.
