## Issues found

1. Medium: restored terminals lose `autoApprove`, so the inheritance chain breaks after workspace/state reload.
   Evidence:
   - `snapshotState()` persists the full terminal object, so `autoApprove` is saved when present: `src/App.tsx:49`.
   - `migrateProjects()` rebuilds terminal objects but only restores `scrollback` and `sessionId`; it drops both `initialPrompt` and the new `autoApprove` field: `src/App.tsx:19`.
   - PTY launch reads `terminal.autoApprove` to decide whether to prepend `--dangerously-skip-permissions` / `--full-auto`: `src/terminal/TerminalTile.tsx:203`, `src/terminal/cliConfig.ts:198`.
   Impact:
   - A Hydra-created sub-agent terminal launched with `--auto-approve` will come back as a normal restricted terminal after restore/reopen, which breaks the advertised end-to-end inheritance path.

2. Medium: `--auto-approve` silently does nothing for Hydra agent types other than Claude/Codex, but the new SKILL guidance is written as a general rule.
   Evidence:
   - Hydra forwards `--auto-approve` for any `--type`: `hydra/src/spawn.ts:88`, `hydra/src/termcanvas.ts:33`.
   - Only the `claude` and `codex` launch configs define `autoApproveArgs`; `kimi`, `gemini`, `opencode`, `tmux`, etc. do not: `src/terminal/cliConfig.ts:66`, `src/terminal/cliConfig.ts:85`, `src/terminal/cliConfig.ts:104`, `src/terminal/cliConfig.ts:123`, `src/terminal/cliConfig.ts:142`, `src/terminal/cliConfig.ts:161`.
   - The skill says “always pass `--auto-approve` so sub-agents inherit the same level of autonomy” without narrowing that claim to supported CLIs: `hydra/skill/SKILL.md:27`.
   Impact:
   - For non-Claude/Codex Hydra agents, users can follow the documented rule and still get a prompting/stalled sub-agent. Either the unsupported types need explicit rejection/warning, or the guidance needs to say inheritance currently exists only for specific CLIs.

3. Low: `hydra init` still installs stale Hydra instructions, so the new permission-inheritance rule will not reach repos that rely on generated `AGENTS.md` / `CLAUDE.md`.
   Evidence:
   - The embedded `HYDRA_SECTION` in `hydra/src/init.ts` contains the old workflow only and no `--auto-approve` guidance: `hydra/src/init.ts:7`.
   - The new rule exists only in `hydra/skill/SKILL.md`: `hydra/skill/SKILL.md:21`.
   Impact:
   - Newly initialized repos will miss the rule entirely, which undermines the documentation change and leaves the “sub-agents stall on approval prompts” problem unsolved for that path.

4. Low: the new privilege bit is accepted and forwarded without any policy gate on the local HTTP API.
   Evidence:
   - The API server listens on `127.0.0.1` and has no auth/origin checks in request handling: `electron/api-server.ts:21`, `electron/api-server.ts:39`.
   - `/terminal/create` now accepts `body.autoApprove` and forwards it directly into renderer state: `electron/api-server.ts:219`.
   - Renderer launch then maps that bit to `--dangerously-skip-permissions` / `--full-auto`: `src/terminal/cliConfig.ts:77`, `src/terminal/cliConfig.ts:96`, `src/terminal/cliConfig.ts:207`.
   Impact:
   - If the threat model does not trust every local process that can reach the TermCanvas API, this change widens what an untrusted localhost caller can request. If localhost is fully trusted, this may be acceptable, but it should be an explicit decision.
