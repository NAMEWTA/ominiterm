## Issues found

### 1. Time-of-day heatmap drops most sessions before aggregation

**Exact code location**

- `electron/insights-shared.ts:600`
- `electron/insights-shared.ts:601`
- `electron/insights-shared.ts:678`
- `electron/insights-shared.ts:691`
- `electron/insights-report.ts:674`

**Root cause**

`aggregateFacets()` builds `facetsById`, then filters `sessions` down to `analyzedSessions`:

```ts
const facetsById = new Map(facets.map((facet) => [facet.session_id, facet]));
const analyzedSessions = sessions.filter((session) => facetsById.has(session.id));
```

Every session-level metric after that, including `messageHourBreakdown`, is aggregated from `analyzedSessions` only:

```ts
for (const session of analyzedSessions) {
  for (const [hour, count] of Object.entries(session.metrics.messageHours)) {
    incr(stats.messageHourBreakdown, hour, count);
  }
}
```

That means any session without a cached/extracted facet is excluded from the heatmap, even though the heatmap is derived from raw session metrics, not facet output. With the current extraction cap in `electron/insights-engine.ts:64-66` and `electron/insights-engine.ts:1482-1486`, the report can easily end up representing only a small recent subset of sessions. If those facet-backed sessions happen to cluster late at night, the heatmap shows activity only at `22:00` and `23:00`.

**Concrete fix proposal**

Split aggregation into:

- session-backed metrics: computed from all eligible `sessions`
- facet-backed classifications: computed from `facets`

```ts
export function aggregateFacets(
  facets: SessionFacet[],
  sessions: SessionInfo[],
  counts: InsightsPipelineCounts,
): AggregatedStats {
  const analyzedSessions = sessions.filter((session) =>
    facets.some((facet) => facet.session_id === session.id),
  );
  const metricSessions = sessions;

  const assistantResponseSeconds = metricSessions.flatMap(
    (session) => session.metrics.assistantResponseSeconds,
  );
  const userReplySeconds = metricSessions.flatMap(
    (session) => session.metrics.userReplySeconds,
  );

  const stats: AggregatedStats = {
    ...counts,
    totalSessions: analyzedSessions.length,
    totalMessages: metricSessions.reduce((sum, session) => sum + session.messageCount, 0),
    totalDurationMinutes: metricSessions.reduce(
      (sum, session) => sum + session.durationMinutes,
      0,
    ),
    averageAssistantResponseSeconds: averageRounded(assistantResponseSeconds),
    averageUserReplySeconds: averageRounded(userReplySeconds),
    // ...
  };

  for (const session of metricSessions) {
    for (const [hour, count] of Object.entries(session.metrics.messageHours)) {
      incr(stats.messageHourBreakdown, hour, count);
    }
    // other session-metric loops stay here
  }

  for (const facet of facets) {
    // facet-backed loops stay here
  }

  return stats;
}
```

If the product wants every number in the report to reflect only facet-backed sessions, then the heatmap should be labeled that way. The current label in `electron/insights-report.ts:91` implies whole-dataset behavior, so the safer fix is to aggregate the heatmap from all eligible sessions.

### 2. All seven AI sections are fed the same data context

**Exact code location**

- `electron/insights-engine.ts:1162`
- `electron/insights-engine.ts:1177`
- `electron/insights-engine.ts:1287`
- `electron/insights-engine.ts:1326`

**Root cause**

`buildAnalysisDataContext()` produces one generic blob containing:

- full `stats`
- the same `sampleFacets` slice

```ts
function buildAnalysisDataContext(stats: InsightsResult["stats"], facets: SessionFacet[]): string {
  const sampleFacets = facets.slice(0, ANALYSIS_SAMPLE_LIMIT).map((facet) => ({
    session_id: facet.session_id,
    underlying_goal: facet.underlying_goal,
    brief_summary: facet.brief_summary,
    project_area: facet.project_area,
    outcome: facet.outcome,
    session_type: facet.session_type,
    notable_tools: facet.notable_tools,
    dominant_languages: facet.dominant_languages,
    wins: facet.wins,
    frictions: facet.frictions,
    recommended_next_step: facet.recommended_next_step,
  }));

  return JSON.stringify({ stats, sampleFacets }, null, 2);
}
```

`runInsightRounds()` computes that once and passes the identical `dataCtx` to every round:

```ts
const dataCtx = buildAnalysisDataContext(stats, facets);

// later
buildAnalysisPrompt(key, dataCtx)
```

The only thing that changes per round is the output schema. The evidence payload does not change, so the model keeps anchoring on the same repeated top-line stats in every section.

**Concrete fix proposal**

Make the context section-specific. The minimal change is to pass `key` into the context builder and tailor both the stats subset and sampled facets to that section.

```ts
function buildAnalysisDataContext(
  key: Exclude<InsightsSectionKey, "atAGlance">,
  stats: InsightsResult["stats"],
  facets: SessionFacet[],
): string {
  switch (key) {
    case "projectAreas":
      return JSON.stringify(
        {
          stats: {
            totalSessions: stats.totalSessions,
            projectAreaBreakdown: stats.projectAreaBreakdown,
            goalCategories: stats.goalCategories,
          },
          sampleFacets: facets.slice(0, ANALYSIS_SAMPLE_LIMIT).map((facet) => ({
            session_id: facet.session_id,
            project_area: facet.project_area,
            underlying_goal: facet.underlying_goal,
            brief_summary: facet.brief_summary,
            outcome: facet.outcome,
          })),
        },
        null,
        2,
      );

    case "whatWorks":
      return JSON.stringify(
        {
          stats: {
            outcomeBreakdown: stats.outcomeBreakdown,
            toolBreakdown: stats.toolBreakdown,
          },
          sampleFacets: facets
            .filter((facet) =>
              facet.outcome === "fully_achieved" || facet.outcome === "mostly_achieved",
            )
            .slice(0, ANALYSIS_SAMPLE_LIMIT)
            .map((facet) => ({
              session_id: facet.session_id,
              underlying_goal: facet.underlying_goal,
              wins: facet.wins,
              notable_tools: facet.notable_tools,
              recommended_next_step: facet.recommended_next_step,
            })),
        },
        null,
        2,
      );

    case "frictionAnalysis":
      return JSON.stringify(
        {
          stats: {
            frictionCounts: stats.frictionCounts,
            toolErrorBreakdown: stats.toolErrorBreakdown,
            responseTimeBreakdown: stats.responseTimeBreakdown,
          },
          sampleFacets: facets
            .filter((facet) => facet.frictions.length > 0 || Object.keys(facet.friction_counts).length > 0)
            .slice(0, ANALYSIS_SAMPLE_LIMIT)
            .map((facet) => ({
              session_id: facet.session_id,
              outcome: facet.outcome,
              frictions: facet.frictions,
              friction_counts: facet.friction_counts,
              brief_summary: facet.brief_summary,
            })),
        },
        null,
        2,
      );

    // add focused cases for the remaining rounds
  }
}
```

Then in `runInsightRounds()`:

```ts
const response = await invokeCli(
  cliSpec,
  analyzerCli,
  buildAnalysisPrompt(key, buildAnalysisDataContext(key, stats, facets)),
);
```

For `funEnding`, the current facet payload is also thin. A stronger version would pass a few raw transcript excerpts from `SessionInfo.analysisText`, which means extending `runInsightRounds()` to receive `sessions` as well.

### 3. `user_satisfaction` is requested without an inference rubric

**Exact code location**

- `electron/insights-engine.ts:1068-1108`
- specifically `electron/insights-engine.ts:1078`
- and the surrounding rules at `electron/insights-engine.ts:1087-1091`

**Root cause**

The facet prompt asks the model to emit:

```ts
'- user_satisfaction: one of "high","medium","low","unclear"',
```

But it never defines how satisfaction should be inferred from:

- explicit user sentiment in the transcript
- outcome quality
- retries/tool failures
- whether the session ended with resolution or churn

So the model has no rubric and defaults to `"unclear"` whenever sentiment is not explicit. This is a prompt-spec bug, not a parsing bug.

**Concrete fix proposal**

Add a decision rubric directly into the prompt rules so the model can infer satisfaction from the same evidence it already receives.

```ts
const prompt = [
  "Analyze this AI coding session and return a JSON object with exactly these fields:",
  // ...
  '- user_satisfaction: one of "high","medium","low","unclear"',
  "",
  "CRITICAL RULES:",
  "- Use the metrics as primary evidence and the transcript excerpt as context.",
  "- Infer user_satisfaction from evidence; do not default to unclear when the session outcome and friction signals are decisive.",
  '- Set user_satisfaction to "high" when the goal was fully or mostly achieved with limited friction, or when the user expresses clear satisfaction/relief.',
  '- Set user_satisfaction to "medium" when the session made useful progress but had notable friction, mixed results, or unresolved follow-up work.',
  '- Set user_satisfaction to "low" when the goal was not achieved, the session shows repeated retries/tool failures/context loss, or the user expresses frustration.',
  '- Set user_satisfaction to "unclear" only when the transcript and metrics do not provide enough signal.',
  "- Consider outcome, friction_counts, tool errors, retries, and closing transcript tone together.",
  "- Do not invent emotional language that is not present.",
  "- Return ONLY valid JSON. No markdown fences. No prose.",
  // ...
].join(\"\\n\");
```

If you want stronger consistency, add a second field like `satisfaction_evidence: string` during extraction, then strip it before final rendering or keep it internal for QA.

### 4. “Analyzed / scanned” mixes incompatible pipeline stages

**Exact code location**

- `electron/insights-engine.ts:932`
- `electron/insights-engine.ts:1457`
- `electron/insights-engine.ts:1537-1545`
- `electron/insights-shared.ts:612`
- `electron/insights-report.ts:103`
- `electron/insights-report.ts:139`
- `electron/insights-report.ts:648`

**Root cause**

The pipeline tracks three different counts:

- raw files scanned: `scanSessions()` returns `totalScannedSessions: files.length` at `electron/insights-engine.ts:932`
- eligible parsed sessions: `generateInsights()` passes `totalEligibleSessions: sessions.length` at `electron/insights-engine.ts:1541`
- facet-backed analyzed sessions: `aggregateFacets()` sets `totalSessions: analyzedSessions.length` at `electron/insights-shared.ts:612`

The report then surfaces those values with compressed wording:

```ts
["Sessions", formatNumber(stats.totalSessions)]
```

```ts
This report analyzed ${stats.totalSessions} sessions out of ${stats.totalEligibleSessions} eligible ...
```

```ts
${stats.totalSessions} analyzed / ${stats.totalScannedSessions} scanned
```

That header is misleading because `scanned` means raw discovered JSONL files, while `analyzed` means only sessions that ended up with facets in this run. Those are different pipeline stages, so the ratio reads like a bug even when it is technically “correct”.

**Concrete fix proposal**

Expose the stage names explicitly and stop compressing them into a single ratio.

```ts
const facetBackedSessions = formatNumber(stats.totalSessions);
const eligibleSessions = formatNumber(stats.totalEligibleSessions);
const scannedFiles = formatNumber(stats.totalScannedSessions);

// header pills
<div class="meta-pill">{facetBackedSessions} facet-backed sessions in report</div>
<div class="meta-pill">{eligibleSessions} eligible sessions parsed</div>
<div class="meta-pill">{scannedFiles} raw session files scanned</div>
```

Update the coverage copy so the cap and deferral mechanism are visible:

```ts
<p class="panel-copy">
  Scanned ${formatNumber(stats.totalScannedSessions)} raw session files and parsed
  ${formatNumber(stats.totalEligibleSessions)} eligible ${escapeHtml(stats.sourceCli)} sessions.
  This report includes ${formatNumber(stats.totalSessions)} sessions with cached or newly extracted facets.
  ${formatNumber(stats.failedFacetSessions)} facet extractions failed and
  ${formatNumber(stats.deferredFacetSessions)} sessions were deferred by the extraction cap.
</p>
```

If you want one compact headline number, make it a same-stage comparison, for example:

```ts
${formatNumber(stats.totalSessions)} facet-backed / ${formatNumber(stats.totalEligibleSessions)} eligible
```

That maps directly to the actual report scope.

## Any unresolved problems

- The proposed fix for section repetition is architectural, not cosmetic. If you keep a single shared data blob for all rounds, the repetition risk will remain even if prompts are reworded.
- The report currently mixes session-backed metrics and facet-backed summaries in one `stats` object. Long term, separate those domains explicitly to avoid future mislabeling bugs.
