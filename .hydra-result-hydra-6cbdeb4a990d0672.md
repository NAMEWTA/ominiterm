## Files changed and why

## Clipboard race conditions identified

### 1. High: the paste flow is not atomic
- The Claude-style paste path snapshots the clipboard, writes new clipboard contents, sends a paste key sequence to the PTY, waits a fixed delay, sends `\r`, then restores the snapshot (`electron/composer-submit.ts:389-473`).
- There is no lock, transaction, or generation check around those steps.
- `ipcMain.handle("composer:submit", ...)` just invokes `submitComposerRequest(...)`; it does not serialize submissions (`electron/main.ts:491-558`).
- On macOS the general pasteboard is a shared system pasteboard, so another app or window can read or overwrite it at any time. Apple describes pasteboards as a shared repository where writers deposit data and readers access it asynchronously, and Electron exposes direct system clipboard reads/writes rather than a transactional API.

Impact:
- Another process can overwrite the clipboard between "write composer payload" and "Claude reads clipboard".
- Another process can overwrite the clipboard between "snapshot" and "restore".
- The submit flow can paste the wrong content into Claude and can also restore stale clipboard contents over a newer user copy.

### 2. High: snapshot itself is not atomic
- `snapshotClipboard()` does `clipboard.readImage()` and `clipboard.readText()` as two separate reads (`electron/main.ts:584-589`).
- If another process changes the clipboard between those two calls, the snapshot can contain mismatched generations, for example old image + new text.

Impact:
- Even the "saved original clipboard" can be internally inconsistent.

### 3. Medium: restore is lossy and not atomic
- `restoreClipboard()` first does `clipboard.clear()`, then writes back only `text` and `image` (`electron/main.ts:592-601`).
- Any clipboard formats other than plain text and image are discarded on restore.
- Because `clear()` and `write(...)` are separate operations, there is a small interleaving window where the clipboard is empty or another writer can race in between them.

Impact:
- Rich text, file URLs, HTML, multiple items, or other formats are not preserved.
- Another app could observe or overwrite the temporary empty clipboard.

### 4. High: user copy/paste actions race directly with submission
- If the user presses `Cmd+C` after TermCanvas snapshots the clipboard but before restore, their new clipboard contents can be overwritten by the old snapshot at the end.
- If the user presses `Cmd+V` in another app during submission, that app can paste the Composer payload instead of the user's own clipboard contents.

Impact:
- This is both a reliability problem for Composer and a direct clipboard-integrity problem for the user.

### 5. High: another TermCanvas window can corrupt the same global clipboard
- The system clipboard is global, but there is no app-wide mutex around paste-mode submissions.
- Two windows can overlap:
  1. Submit A snapshots `X`, writes payload `A`, triggers paste, waits.
  2. Submit B snapshots `A`, writes payload `B`, triggers paste, waits.
  3. Submit A restores `X` while B is still in flight.
  4. Submit B restores `A` at the end.

Impact:
- Claude can receive `X`, `A`, or `B` depending on timing.
- The final clipboard can end up as `A`, not the user's original `X`.

## Timing analysis

### 120ms in the current code
- `pasteDelayMs` is hardcoded in terminal config (`src/terminal/cliConfig.ts:17-26`).
- For Claude/Kimi/Gemini/OpenCode paste mode it is fixed at `120` ms (`src/terminal/cliConfig.ts:82-160`).
- It is not user-configurable at runtime in the current codebase.
- `ptyManager.write(...)` is fire-and-forget and provides no acknowledgment that the target app has handled the paste key or already read the clipboard (`electron/pty-manager.ts:38-40`).

### What happens during the 120ms
- TermCanvas is waiting after sending the paste shortcut so the target CLI has time to:
  1. receive the PTY key sequence,
  2. interpret it as "paste from clipboard",
  3. read the system clipboard,
  4. insert the text/image/path into its UI state.
- There is no signal back from Claude Code proving those steps completed before TermCanvas continues.

### Is 120ms enough?
- It is a heuristic, not a guarantee.
- Clipboard write to the macOS pasteboard is usually fast, but the dominant latency is likely the target app's input/event-loop scheduling and clipboard read timing, not the raw clipboard write itself.
- If the target CLI is busy, blocked on rendering, or otherwise delayed, 120ms can be too short.
- Because TermCanvas may overwrite the clipboard again after the timer expires, a slow paste consumer can read the next payload or the restored snapshot instead of the intended one.

Assessment:
- For intermittent failures, 120ms is not defensible as a correctness boundary.
- It may be sufficient on an idle machine, but it is not sufficient by construction under load because there is no acknowledgment path.

### Multiple images increase the risk window
- `submitImages(...)` waits `pasteDelayMs` after every image paste (`electron/composer-submit.ts:168-210`).
- The text paste then waits another `pasteDelayMs` before Enter (`electron/composer-submit.ts:408-432`).
- Total clipboard-hijack window is therefore roughly:
  - images only: `imageCount * 120ms`
  - images + text: `(imageCount + 1) * 120ms`
- Example: 4 images + text keeps the global clipboard commandeered for at least about `600ms`, plus the actual write and PTY handling overhead.

Impact:
- More images means more opportunities for external clipboard writes to interrupt the flow.
- More images also make it more likely that the target app has not consumed image `N` before TermCanvas overwrites the clipboard with image `N+1` or the final text payload.

## Clipboard snapshot/restore analysis

### If snapshot fails
- `snapshotClipboardSafely(...)` converts exceptions into a `"clipboard-capture-failed"` issue (`electron/composer-submit.ts:154-166`).
- `submitComposerRequest(...)` returns immediately if snapshot fails (`electron/composer-submit.ts:389-392`).

Impact:
- Submission fails cleanly before any clipboard mutation.
- This is not a clipboard leak path.

### If restore fails
- The `finally` block always attempts `deps.restoreClipboard(snapshot)` after snapshot succeeds (`electron/composer-submit.ts:457-470`).
- On the normal success/failure paths, a restore failure is surfaced as warning code `"clipboard-restore-failed"`.
- Existing tests cover the success-path warning (`tests/composer-submit.test.ts`).

Impact:
- The clipboard may remain as the Composer payload if restore fails.
- There is no retry and no fallback strategy.

### Race with user activity between snapshot and restore
- Yes. This is the core design problem.
- The code never checks whether the clipboard still contains the Composer payload before restoring.
- If the user intentionally copied something new during submission, TermCanvas will still restore the old snapshot and destroy the user's newer clipboard.

## Concurrent submission analysis

### Renderer-side guard is not sufficient
- `ComposerBar.handleSubmit` does not start with `if (isSubmitting) return;` (`src/components/ComposerBar.tsx:214-287`).
- It calls `setSubmitting(true)` only after building the request, then awaits the IPC call (`src/components/ComposerBar.tsx:251-263`).
- The UI disables the textarea and button based on `isSubmitting`, but that only takes effect after React re-renders (`src/components/ComposerBar.tsx:399-415`).

Impact:
- Two rapid Enter key events or two rapid clicks can call `handleSubmit()` twice before the disabled state is reflected in the DOM.
- The code relies on UI timing instead of an actual re-entrancy guard.

### Store-side guard is also not sufficient
- `composerStore` only has `setSubmitting(isSubmitting)`; it is not a compare-and-set or mutex (`src/stores/composerStore.ts:18-43`).
- Nothing prevents concurrent callers from setting `true`, then both continuing into separate IPC submissions.

### Main process does not serialize submissions
- `ipcRenderer.invoke("composer:submit", request)` is a plain async call (`electron/preload.ts:121-124`).
- `ipcMain.handle("composer:submit", ...)` does not enforce one-at-a-time submission globally, per PTY, or per terminal (`electron/main.ts:491-558`).

Assessment:
- Concurrent submission is possible.
- Because paste mode uses the global clipboard, concurrent submission is a high-severity reliability issue even if it occurs only occasionally.

## Early-return clipboard leak analysis

### The suspected `return` at lines 412-420 does not skip restore
- The text clipboard write failure path returns from inside the `try` block (`electron/composer-submit.ts:409-420`).
- In JavaScript, `finally` still runs when a `try` block returns.
- So `deps.restoreClipboard(snapshot)` is still attempted on that path.

Conclusion:
- The specific claim that this path returns "without restoring clipboard" is incorrect.

### Actual early-return paths
- Before snapshot:
  - unsupported terminal (`electron/composer-submit.ts:336-343`)
  - empty submit (`electron/composer-submit.ts:345-351`)
  - unsupported images (`electron/composer-submit.ts:361-367`)
  - image staging failure (`electron/composer-submit.ts:371-387`)
  - snapshot failure (`electron/composer-submit.ts:389-392`)
- These paths happen before any clipboard mutation, so they do not leak the user's clipboard.

### Real remaining issue on the 412-420 path
- If restore itself throws during that return-from-try path, the restore warning is unlikely to be reflected in the returned failure object, because the function is already returning a value directly from inside the `try`.
- So the clipboard restore is attempted, but a simultaneous restore failure may be underreported to the caller.

## Specific reproducible scenarios

### Scenario 1: user copies something during submit
1. User clipboard starts as `X`.
2. Composer snapshots `X`.
3. Composer writes message `M` to clipboard and triggers Claude paste.
4. Before Claude reads the clipboard, user presses `Cmd+C` on `Y`.
5. Clipboard becomes `Y`.
6. Claude may paste `Y` instead of `M`.
7. Composer restore writes old snapshot `X`, destroying the user's newer clipboard `Y`.

Severity: High

### Scenario 2: user pastes in another app during submit
1. Composer writes its payload to the system clipboard.
2. User quickly switches apps and presses `Cmd+V`.
3. The other app pastes the Composer text/image/path.

Severity: High

### Scenario 3: two TermCanvas windows submit concurrently
1. Window A snapshots `X`, writes payload `A`, triggers paste.
2. Window B snapshots `A`, writes payload `B`, triggers paste.
3. A restores `X` before B has finished consuming `B`.
4. B restores `A` at the end.

Possible outcomes:
- Claude in A or B receives the wrong payload.
- Final clipboard becomes `A`, not the user's original `X`.

Severity: High

### Scenario 4: multi-image submit on a busy machine
1. Image 1 is written to clipboard, paste is triggered, 120ms elapses.
2. Claude has not consumed image 1 yet because it is busy.
3. Composer overwrites clipboard with image 2.
4. Claude reads image 2 for both paste events, or misses image 1 entirely.

Severity: High

### Scenario 5: clipboard manager or other app rewrites the clipboard
1. Composer writes payload `M`.
2. A clipboard manager or another app normalizes/transforms/replaces the clipboard contents.
3. Claude reads the transformed value, or TermCanvas restores an old snapshot over a new external write.

Severity: Medium to High

## Overall conclusion
- The intermittent failure is plausibly explained by the current design.
- The core issue is not just "120ms might be too short"; it is that the design depends on a global, shared clipboard plus a blind timer and has no acknowledgment, generation check, or submission mutex.
- 120ms can make the failure more likely, but even a longer delay would still leave the design race-prone.
